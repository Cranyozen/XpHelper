## 二、项目背景与动机

在 Minecraft 原版生存服务器中，经验值（XP）系统存在以下痛点：

1. **经验不可直接操控** — 原版仅提供 `xp` 命令的简单加减，缺少批量转移、按比例操作等高级功能。
2. **经验修补效率低** — 玩家需要手动站到经验球旁边等待修补，无法批量或自动处理。
3. **管理工具缺失** — 服务器管理员缺少统一的经验管理界面，无法便捷地帮助玩家处理经验相关事务。
4. **信息不透明** — 玩家难以准确了解自己经验值的详细构成（点数、等级、装备修复需求等）。

Xp Helper 旨在通过 MCDReforged 插件框架，为服务器提供一套完整的经验值管理方案，填补原版命令的功能空白。

```
MCDReforged >= 2.6.0     (插件框架)
minecraft_data_api >= 1.5.0  (NBT 数据解析)
sympy >= 1.12             (符号数学计算)
```

### 4.4 通信机制

```
插件 ←→ MCDReforged API ←→ Minecraft 服务器
  │                              │
  │   (server.execute)           │ (RCon 协议)
  │                              │
  └────────── RCon ←─────────────┘
         (获取命令返回结果)
```

插件使用 RCon 协议获取命令执行结果（如玩家列表、经验查询、NBT 数据），这是因为 MCDR 的 `server.execute()` 只能推送命令而无法获取返回值。

## 五、模块详细设计

### 5.1 命令系统（命令树）

所有命令以 `!!xp` 为根前缀：

```
!!xp                                    → 帮助信息
├── info [<玩家>]                       → 经验信息查询
├── move <目标> [<来源>] <数量> [SURE]  → 经验转移
├── drop [<玩家>] (all|<数量>|<比率>) [SURE] → 经验球化
├── repair [<玩家>] (all|<比率>|auto [<布尔>]) [SURE] → 装备修复
└── reload                              → 配置重载
```

**权限模型**：
- **普通玩家**：操作自身经验的全部功能
- **助手/管理员**（默认权限等级 ≥ 2）：可操作任意玩家
- **控制台**：不能执行需要玩家上下文的命令

### 5.2 config.py — 配置系统

使用 MCDR `Serializable` 实现声明式配置，自动从 `config/xp_helper.json` 加载/保存。

**配置项清单**：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `repair_max_time` | int | 5 | 单次修复最大持续秒数 |
| `repair_interval` | float | 0.2 | 修复循环中召唤经验球的间隔（秒） |
| `repair_max_point_rate` | float | 1.05 | 按比例修复时最大经验点倍率 |
| `repair_rate` | float | 1.05 | 计算所需经验时对耐久的放大系数 |
| `auto_repair_player_list` | list[str] | [] | 自动修复玩家名单 |
| `auto_repair_interval` | int | 60 | 自动修复检查间隔（秒） |
| `mending_rate` | float | 0.5 | 每点耐久所需经验点（经验修补附魔系数） |
| `orb_value` | list[int] | [149,73,37,17,7,3,1] | 分批召唤时经验球价值分档 |
| `summon_interval` | float | 0.1 | 分批召唤时每个球间隔（秒） |
| `helper_permission` | int | 2 | 管理员操作所需最低权限等级 |
| `summon_orb_direct` | bool | True | 是否直接召唤总价值经验球 |

### 5.3 handles/xp_info.py — 经验信息查询

**职责**：查询并展示玩家的经验与装备修复信息。

**数据流**：
```
玩家触发 !!xp info
    → get_player_xp() 获取总经验点数
    → get_player_durability_value() 获取装备耐久损耗
    → xp_convert 点数→等级换算
    → 计算修复所需经验
    → 返回格式化信息（点数/等级/耐久/修复预估）
```

**输出信息包含**：
- 总经验点数 & 换算等级
- 所有可修复装备的总耐久损耗
- 修复全部装备所需经验点数
- 经验是否充足/不足的评估

### 5.4 handles/xp_drop.py — 经验球化

**职责**：扣除玩家经验并召唤对应价值的经验球。

**数据流**：
```
玩家触发 !!xp drop all
    → ask_for_sure 二次确认
    → get_player_xp() 获取当前经验
    → remove_player_xp() 扣除经验
    → summon_orb() 召唤经验球
```

**召唤模式**（配置可选）：
1. **直接模式** (`summon_orb_direct=true`)：一次召唤总价值经验球，使用 `execute at <player> run summon minecraft:experience_orb`
2. **分批模式** (`summon_orb_direct=false`)：按 `orb_value` 分档分解价值，逐批召唤，间隔 `summon_interval` 秒

**参数解析**：
- `all`：全部经验
- 整数（如 `100`）：指定点数
- 小数（如 `0.5`）：总经验的百分比

### 5.5 handles/xp_move.py — 经验转移

**职责**：在玩家之间转移经验值。

**数据流**：
```
玩家触发 !!xp move <目标> <数量> [SURE]
    → 验证目标在线且不是自己
    → ask_for_sure 二次确认
    → remove_player_xp(源) 扣除
    → add_player_xp(目标) 增加
    → 确认成功 → 通知双方
```

**安全措施**：
- 源和目标不能为同一玩家
- 双方必须在线
- 必须有足够的经验
- 需要二次确认

### 5.6 handles/xp_repair.py — 装备修复（最复杂模块）

#### 5.6.1 手动修复

独立线程运行，不阻塞服务器主循环。

```
玩家触发 !!xp repair all
    → ask_for_sure 二次确认
    → 启动修复线程 (max repair_max_time 秒)
       循环:
         ├── get_player_repair_durability_value() 获取当前耐久
         ├── 计算所需经验点数
         ├── 如果经验不足 → 中断并通知
         ├── 如果所有装备已修好 → 中断并通知
         ├── xp_drop_handle() 扣经验并召唤经验球
         └── 等待 repair_interval 秒
    → 结束（超时或条件中断）
```

#### 5.6.2 自动修复

全局独立后台线程，与服务器生命周期同步。

```
线程启动 (on_server_startup / on_load):
    无限循环:
      ├── 等待 auto_repair_interval 秒
      ├── 获取所有在线玩家
      ├── 只处理在 auto_repair_player_list 中的玩家
      ├── 执行修复逻辑（同手动修复）
      └── 通过 server.tell() 发送通知
```

**状态管理**：玩家可通过 `!!xp repair auto true/false` 开关自动修复，状态变更后自动保存配置。

#### 5.7.2 player_xp.py — 经验操作

| 函数 | 说明 |
|------|------|
| `get_player_xp(src, player)` | 查询玩家总经验点数 |
| `remove_player_xp(src, player, amount)` | 扣除玩家经验 |
| `add_player_xp(src, player, amount)` | 增加玩家经验 |
| `move_player_xp(src, target, source, amount)` | 原子化经验转移 |

使用 `xp query` / `xp add` 命令操作，通过 RCon 获取命令返回结果。

#### 5.7.3 player_item.py — 物品查询

| 函数 | 说明 |
|------|------|
| `get_player_all_item(src, player)` | 获取玩家全部背包物品 |
| `get_player_durability_item(src, player)` | 筛选可修复且有耐久的物品 |
| `get_player_durability_value(src, player)` | 计算所有带经验修补的装备的耐久损耗 |
| `get_player_repair_durability_value(src, player)` | 仅计算主手+盔甲栏的耐久损耗 |

**盔甲栏位**：`100` (头盔) / `101` (胸甲) / `102` (护腿) / `103` (靴子) / `-106` (副手)

#### 5.7.4 item_durability.py — 耐久数据库

完整内置的 Minecraft 1.20.1 物品耐久数据库，涵盖：

- **盔甲类**：海龟壳、皮革、金、锁链、铁、钻石、下界合金 × 各部位
- **工具类**：木/石/铁/金/钻石/下界合金的剑、镐、斧、锹、锄
- **特殊工具**：弓、弩、三叉戟、鞘翅、盾牌、钓鱼竿、剪刀、打火石、胡萝卜钓竿、诡异菌钓竿、刷子、山羊角
- **头盔类**：骷髅头颅、凋零骷髅头颅、龙首、苦力怕头颅、玩家头颅、猪灵头颅

**关键函数**：
- `get_durability_by_id(item_id)`：通过物品 ID 查找总耐久
- `get_require_point_by_id(item_id, current_durability)`：计算修复所需经验
- `get_require_point_by_durability(durability)`：通过损耗值计算经验（公式：`耐久损耗 × 经验修补系数 + 1`）
- `check_item(item_id)`：判断物品是否可修复

**ID 解析**：按 `_` 分割物品 ID，匹配 `材质_部位` 或 `特殊物品` 模式。

#### 5.7.5 xp_convert.py — 等级↔点数转换

实现 Minecraft 官方经验公式的正向与逆向计算。

**正向（等级→点数）**：

| 等级范围 | 公式 |
|----------|------|
| 0~16 | `等级² + 6×等级` |
| 17~31 | `2.5×等级² - 40.5×等级 + 360` |
| 32+ | `4.5×等级² - 162.5×等级 + 2220` |

**逆向（点数→等级）**：使用 sympy 解二次方程，取非负实数解。

**临界值**：352 点 = 16 级，1507 点 = 31 级。

#### 5.7.6 xp_orb.py — 经验球召唤

```
直接模式:
    execute at <玩家> run summon minecraft:experience_orb ~ ~1 ~ {Value:<总价值>}

分批模式 (按 [149,73,37,17,7,3,1] 分档):
    for value in 分档值:
        execute at <玩家> run summon minecraft:experience_orb ~ ~1 ~ {Value:<value>}
        wait summon_interval 秒
```

分批模式类似找零算法，将总价值按最大面额分解，避免单个经验球价值过高。

## 六、安全设计

### 6.1 风险分析与防护

| 风险场景 | 防护措施 |
|----------|----------|
| 误操作消耗经验 | 所有消耗性操作需追加 `SURE` 二次确认 |
| 普通玩家越权操作他人 | 操作他人需 `helper_permission` 权限检查（默认 2 级） |
| 控制台执行玩家命令 | 每条命令前检查调用者身份 |
| 操作离线玩家 | 执行前验证目标玩家在线 |
| 自己转给自己 | 源和目标玩家 ID 比对 |
| 经验不足时仍执行 | 操作前读取当前经验值做容量检查 |
| 修复线程无限运行 | `repair_max_time` 硬性时间上限 |
| 配置损坏导致插件崩 | MCDR `Serializable` 内置验证与异常保护 |

### 6.2 二次确认机制

```
玩家: !!xp drop all
插件: §a[确认] 你确定要丢弃所有经验吗？点击确认或输入 !!xp drop all SURE
玩家: !!xp drop all SURE
插件: §a已丢弃 xxx 经验值
```

通过 RText 给玩家生成可点击的确认按钮，控制台则显示完整命令提示。

## 八、开发与扩展指南

### 8.1 添加新命令

1. 在 `handles/` 下创建新的处理模块
2. 在 `main.py` 的 `register_command` 函数中注册命令树
3. 在语言文件中添加对应的翻译条目
4. 在帮助系统中添加命令说明

### 8.2 添加新配置项

1. 在 `config.py` 的 `XpHelperConfig` 类中添加 `Serializable` 字段
2. 设置合理的默认值
3. 在需要使用配置的地方导入 `config`

### 8.3 扩展物品耐久数据库

在 `item_durability.py` 的 `item_durability_map` 字典中添加新条目，格式为：

```python
"物品id": 总耐久值
```
